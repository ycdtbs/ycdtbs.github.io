2、ThreadPoolExecutor
  • 线程池的状态

ThreadPoolExcutor使用一个整数位来表示线程数量和线程状态，高3为表示线程状态，低29位表示线程数量为什么这么做呢?这样就可以保证每次更新线程状态和线程数量的时候只需要进行一次CAS操作

线程池的五种状态

  • RUNNING 正常启动状态
  • SHUTDOWN 温和关闭，执行完成阻塞队列中的任务
  • STOP 关闭，直接抛弃任务
  • TIDUYING 任务执行完毕，活跃线程为0即将关闭的状态
  • TERMINATED 终结状态

构造方法

  • int cornPoolSize 核心线程数
  • int maxiumPoolSize 最大线程数
  • long keepAlibeTime 救急线程存活时间
  • TimeUnit timeunit 时间单位
  • BlockQuene workQuene 阻塞队列
  • ThreadFactory threadFactiry 线程工厂
  • RejectedExectutionHandler handelr 拒绝策略

当阻塞队列也满了的时候，会创建一个救急线程来救急，救急线程干完活后，到了救济线程时间后就会销毁

线程池运行流程

1、线程池开始的时候并没有线程，当任务交给线程池的时候，线程开始运行

2、当线程池的cornsiz满了的时候会进入到阻塞队列

3、如果选择了有界队列，那么当阻塞队列满了之后会创建最大线程数-核心线程数的救急线程来执行任务

4、如果救急线程仍然无法处理请求，那么这个时候就要执行拒绝策略

5、ThreaPoolExectur给我们提供了四种拒绝策略

  • AbortPolicy 抛出RejectExcutionException
  • CallerRunPlicy 让调用者自己运行
  • DiscardPolicy 放弃任务
  • DiscardOldestPolicy 放弃队列中最早的任务本任务取而代之
  • Dubbo的实现，抛出日志
  • Netty实现 创建新的线程池
  • Activemq实现超时等待
  • PinPoint 使用了拒绝策略链

3、Executors
4、newFixedThreadPool

固定大小的线程池，适合线程数固定，执行任务长

  • 创建固定大小的线程池参数只有一个核心线程数目，没有救急线程，同时阻塞队列使用的LinkBlockQuene
  • 执行完成任务后不会结束程序
  • 默认的工厂方法
5、newCatchThreadPool
  • 无参，没有核心线程，全都是救急线程意味着
    • 救急线程可以无限创建
    • 每个线程存活60s
  • 阻塞队列是SynchronousQueue同步队列
    • 阻塞队列类似于一手交钱一手交货，比如有人取才可以有人放
  • 适用于任务数量多，执行时间短的任务
6、newSingleThreadExecutor
  • 线程池中只有一个线程，固定数为1，会放入无界队列排队，任务执行完毕后唯一的线程也不会被释放

为什么不自己创建一个线程呢

  • 自己创建线程执行后失败了就不会往下继续执行了，此时就i没有别的线程能够执行接下来的任务了，但是如果使用了单线程的线程池可以帮助我们创建一个新的线程来执行任务
  • 和固定大小的线程池有什么区别呢
    • 固定大小的线程池返回值是线程池对象
    • 单例的返回值是使用装饰器模式返回的线程池中的基本方法，限制返回对象暴漏的方法
7、线程池API-提交任务
  • execute

执行一个任务，输入参数是Runable接口

  • submit
  • invokeAll 执行一组线程
  • invokeAny 执行一组线程
    • 找到一个最先执行完的任务
  • shutdown 将线程池状态变为shutdown 会执行完成任务
    • shutdown不会阻塞线程
    • shudown不会接收新任务，但是已经执行的任务会执行完毕
  • shutdownNow
    • 不会接收新任务
    • 会将队列中的任务返回
    • 用interput的方式中断正在执行的任务
  • 其他API
    • isShutdown（） 查看状态
    • isTerminated
    • awaitTermination（long timeout,Timeunit timeUnit）

9、设置多少个线程
  • CPU密集型运算
    • 线程设置为CPU核数 + 1
  • IO密集型运算
    • CPU不总是繁忙状态的例如执行业务计算的时候需要CPU，但是当执行远程方法，rpc调用时候包括操纵数据库的时候，就闲下来了
    • 经验公式如下
    • 线程数 = 核数 * 期望利用率 *总时间/cpu计算时间

11、可调度任务的线程池 newScheduledThreadPool

newScheduledThreadPool

  • 一个任务没有正确的处理异常会不会影响第二个任务？

    • 两个任务都可以正常运行，但是无法捕获到线程池当中的异常

  • 定时执行任务

    • 每隔一秒执行一次 scheduleAtFixedRate scheduleWithFixedDelay
    • 如果每个任务执行时间较长，会影响到现有的线程

  • scheduleAtFixedRate和scheduleWithFixedDelay的区别

    • scheduleAtFixedRate是过了一个period后上一个线程执行完毕后立刻执行
    • scheduleWithFixedDelay是上一个线程结束后过了一个delay在执行
